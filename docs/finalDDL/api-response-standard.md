# BaseResponse (숫자 코드 버전)
## 공통 응답 구조
- HTTP Status Code는 헤더에서만 사용
- BaseResponse.code는 비즈니스 전용 커스텀 코드
- HTTP와 BaseResponse.code는 분리

## 계층 구조
```text
HTTP Status (Header)
    └─ 200 / 400 / 401 / 403 / 404 / 409 / 500

Body
    └─ BaseResponse.code (5자리 비즈니스 코드)
```


## BaseResponse 구조
```json
header
{
  "status": 200
}
body
{
  "code": 20015,
  "message": "참여가 완료되었습니다.",
  "timestamp": "2026-03-03T17:20:00Z",
  "data": { ... }
}
```
## 코드 규칙
### 구조
```text
ABCDE
││││└─ 세부 코드 (00~99)
││└── HTTP 범주 (200 / 400 / 500 등)
```

### 예:
- 20015 → HTTP 200 범주 + 세부 15
- 40901 → HTTP 409 범주 + 세부 01
- 50002 → HTTP 500 범주 + 세부 02



## 코드 체계 설계안 (권장)

HTTP Status Code == BaseResponse.code

data 내부에 주제별(custom) 비즈니스 코드 포함

즉, 계층은 이렇게 갑니다:

```text
HTTP Status Code  →  BaseResponse.code
                    →  data.businessCode (주제별 커스텀 코드)
```

## BaseResponse.code 세부 정의
### 1️⃣ HTTP Status (표준 의미 유지)
HTTP | 의미
------|------
200 | 요청 정상 처리
201 | 생성 성공

403 | 권한 오류
404 | 리소스 없음
409 | 상태 충돌
500 | 서버 오류

## 4️⃣ HTTP 500 Internal Server Error + BaseResponse.code

이 구조는 서버 오류 시에도 클라이언트가 비즈니스 맥락을 이해할 수 있게 해줍니다.

예시:

### HTTP 500 + BaseResponse.code = 50002

→ "외부 API 오류로 인해 처리에 실패했습니다."

### HTTP 500 + BaseResponse.code = 50001

→ "데이터베이스 오류로 인해 처리에 실패했습니다."

## 5️⃣ 장점

- HTTP 상태 코드는 표준 의미 유지
- 클라이언트가 HTTP 상태만 보고도 대략적인 오류 종류 파악 가능
- BaseResponse.code로 비즈니스 로직 오류 구분
- API 문서화 용이
- 코드 체계가 명확하고 확장 가능

## 6️⃣ 권장사항

- BaseResponse.code는 5자리 숫자로 구성
  - 앞 2자리: 주제별 코드 (200=성공, 400=클라이언트 오류, 500=서버 오류 등)
  - 뒤 3자리: 비즈니스 로직 코드
- 주제별로 코드 범위를 미리 할당
  - 예: 20000~29999 = 성공, 40000~49999 = 클라이언트 오류 등
- 문서에 코드 체계표 포함
- 모든 API 응답에 BaseResponse 사용
- HTTP와 BaseResponse.code를 분리하여 사용

## 7️⃣ 요약

- HTTP Status Code: 헤더에서만 사용, 표준 의미 유지
- BaseResponse.code: 본문에서 사용, 비즈니스 전용 커스텀 코드
- HTTP와 BaseResponse.code는 분리
- 이 구조를 사용하면 HTTP 표준을 준수하면서도 비즈니스 로직을 명확하게 표현할 수 있습니다.
